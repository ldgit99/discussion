# AI 기반 공동 토의 지원 에이전트 — 개발 리서치

## 1. 프로젝트 개요

### 1.1 시스템 정의
중학생의 문제 해결 토의를 지원하기 위해 **2-5명의 학생**이 하나의 온라인 토의방에 접속하여, AI의 안내에 따라 돌아가며 자신의 의견과 근거를 제시하고, **개인 채팅(AI와 1:1 사고 확장)** 과 **팀 채팅(공동 토의)** 두 모드를 오가며 의견을 비교·조정하고, 모둠 대표 의견을 합의하도록 돕는 AI 기반 공동 토의 지원 에이전트.

### 1.2 핵심 가치 명제 (Value Proposition)
- **AI가 학생의 사고와 판단을 대체하지 않는다.**
- **학생 간 상호작용과 공동 의사결정을 촉진한다.**
- **Human-AI Agency 기반의 토의 환경을 구현한다.**

### 1.3 사용 맥락
| 단계 | 주체 | 역할 |
|------|------|------|
| 토의 준비 | 교사 | 주제 설정, 사전 지도, 자료 안내 |
| 토의 실행 | 학생 2-5인 + AI | 개인 사고 확장, 의견 공유, 비교·조정, 합의 도출 |
| 토의 모니터링 | 교사 | 대시보드로 모둠별 진행 상황 확인 및 개입 |
| 토의 평가 | 교사 | 최종 교육적 판단 및 수업 조율 |

## 2. 이론적 배경

### 2.1 Human-AI Agency 개념
- **Agency**: 자신의 행동을 의도적으로 선택하고 그 결과에 책임지는 능력.
- **Human-AI Agency**: 인간과 AI가 각자의 행위 주체성을 보존한 채 협력하여 의사결정을 수행하는 구조.
- 본 시스템에서의 적용:
  - 학생의 **사고·판단·결정 권한**은 보호.
  - AI는 **메타인지적 비계(scaffolding)** 와 **상호작용 촉진자(facilitator)** 역할만 수행.
  - 최종 교육적 판단은 **교사**가 보유.

### 2.2 협력적 학습(Collaborative Learning) 이론 기반
- **사회적 구성주의(Vygotsky)**: ZPD(근접발달영역) 내에서 또래 및 도구의 매개를 통한 학습.
- **인지적 갈등(Piaget)**: 의견 차이를 통해 사고가 정교화되는 과정.
- **공동 의미 구성(Joint Meaning-Making)**: 토의 보드를 통한 외현화된 사고 공유.

### 2.3 중학생 인지·사회적 발달 특성
- **형식적 조작기 진입기**: 추상적 사고와 가설 검증이 가능하나 아직 불안정.
- **또래 영향력 증대**: 동료의 의견에 동조하거나 반발하는 경향이 강함.
- **자아 정체성 형성기**: 의견 표현에 대한 자의식이 높음 → 심리적 안전감 설계 필요.
- **메타인지 발달 중**: 자신의 사고 과정을 모니터링하는 능력이 미성숙 → AI 비계 효과적.

### 2.4 관련 선행 연구 키워드
- Computer-Supported Collaborative Learning (CSCL)
- AI in Education (AIED) / Conversational Agents for Learning
- Group Awareness Tools
- Argumentation Scaffolding
- Consensus-Building in Online Discussion

## 3. 수업 흐름 (Class Flow)

본 시스템이 지원하는 전체 수업 흐름은 다음과 같다. **개인 채팅(AI 1:1)** 과 **팀 채팅(모둠 공동)** 두 모드를 오가는 구조가 핵심이다.

```
[1] 교사 주도 준비
    - 토의 주제 설정 (시스템에 등록)
    - 역할과 태도 지도
    - 방 코드 학생 배포
              ↓
[2] 2-5명 토의방 입장
    - 방 코드 입력
    - 닉네임 설정
              ↓
[3] AI 주도 시작 (팀 채팅)
    - AI가 토의 주제 안내
    - AI가 토의 절차 안내 (턴 순서, 시간)
    - 모든 학생에게 동일한 컨텍스트 제공
              ↓
[4] 턴 기반 의견 제시 (팀 채팅)
    - AI 진행자: "○○님 차례입니다. 의견을 말해주세요."
    - 학생 의견 발화
    - AI: 의견에 대한 근거 요구 ("왜 그렇게 생각하나요?")
              ↓
[5] 개인 모드 — 사고 확장 (개인 채팅)
    - 학생이 개인 채팅 탭으로 이동
    - AI와 1:1로 자신의 생각을 확장·정리
    - AI는 명료화·반론 자극·근거 보완 질문 제공
    - 팀에 공유할지 학생이 결정
              ↓
[6] 팀 모드 — 결과 공유 (팀 채팅)
    - 학생이 개인 모드에서 정리한 내용을 팀 채팅에 공유
    - 다른 학생들의 질문과 보완
    - AI: 의견 요약, 비교 기준 제시, 근거 점검, 합의 질문
    - (4→5→6은 학생 수만큼, 또는 합의될 때까지 반복)
              ↓
[7] 모둠 대표 의견 선정 (팀 채팅)
    - AI 합의 보조: 공통점·남은 쟁점 정리
    - 학생들이 직접 대표 의견 작성·확정
              ↓
[8] 결과 저장 및 제출
              ↓
[9] 교사용 대시보드 확인
              ↓
[10] 최종 토의 또는 전체 공유
```

### 채팅 모드 구분

| 모드 | 참여자 | 목적 | AI 역할 |
|------|--------|------|---------|
| **팀 채팅** | 모든 모둠원 + AI | 공동 토의, 의견 공유, 합의 도출 | 진행자 + 보조자 (요약·비교·합의) |
| **개인 채팅** | 학생 1인 + AI | 자기 생각 확장·정리 | 1:1 코치 (명료화·반론·근거 보완) |

**전환 규칙**:
- 학생은 언제든 두 모드를 자유롭게 오갈 수 있다.
- 개인 채팅 내용은 **학생 본인과 교사**만 열람 가능. 팀에 공유하려면 학생이 명시적으로 "팀에 공유" 액션을 수행해야 한다.
- AI 진행자는 팀 채팅의 흐름을 주도하되, 개인 채팅으로의 전환을 권유할 수 있다 ("잠시 혼자 생각을 정리해볼까요?").

### 단계별 시스템 책임 분담

| 단계 | 교사 | 학생 | AI |
|------|------|------|----|
| 1. 준비 | **주도** (주제 등록) | — | 미개입 |
| 2. 입장 | 모니터링 | 방 코드 입력, 닉네임 설정 | 환영 |
| 3. AI 주도 시작 | 모니터링 | 안내 청취 | **주도** (주제·절차 안내) |
| 4. 턴 기반 의견 제시 | 모니터링 | 의견 발화 | **진행** (턴 관리, 근거 요구) |
| 5. 개인 모드 사고 확장 | 비노출 (학생 사고 보호) | 1:1 대화 | 1:1 코치 |
| 6. 팀 모드 결과 공유 | 모니터링/개입 | 발화 주도 | 보조 (요약·비교·합의 질문) |
| 7. 대표 의견 선정 | 모니터링 | **결정 주체** | 합의 보조 (결정 금지) |
| 8. 저장 및 제출 | 검토 | 확정 | 산출물 정리 보조 |
| 9. 대시보드 | **종합 판단** | — | 데이터 집계 |
| 10. 전체 공유 | 진행 | 발표 | 미개입 |

## 4. 시스템 구성 요소

### 4.1 학생 인터페이스
| 영역 | 기능 |
|------|------|
| 토의방 입장 | 방 코드 입력, 닉네임 설정 (2-5명) |
| 팀 채팅 | 모둠 전체 + AI 진행자의 공동 토의 |
| 개인 채팅 | 학생 1인 ↔ AI 1:1 사고 확장 (비공개) |
| 공동 토의 보드 | 의견 카드 시각화, 비교/그룹핑 |
| AI 보조 패널 | 6개 기능 버튼 호출 |
| 합의 영역 | 모둠 대표 의견 작성 및 확정 |

### 4.2 학생용 대시보드 레이아웃

```
┌────────────────────────────────────────────────────────────────────┐
│ 상단 헤더                                                           │
│ 토의 주제 | 현재 단계 | 남은 시간 | 방 코드 | n/N 접속 | 교사명      │
├──────────────────────┬─────────────────────────┬──────────────────┤
│ 채팅 영역             │ 공동 토의 보드            │ AI 보조 패널      │
│ ┌────┬────┐          │                         │                  │
│ │ 팀  │ 개인│  탭 전환  │ 의견 비교표              │ [정리하기]        │
│ └────┴────┘          │ 선택 기준                │ [의견 비교]       │
│                      │ 쟁점·공통점·차이점         │ [근거 확인]       │
│ [팀 모드]             │ 모둠 대표 의견            │ [질문 만들기]     │
│  - AI 진행자 메시지    │ 선택 이유                │ [태도 점검]       │
│  - 학생 발화 (닉네임)  │ 실행 방안                │ [합의 돕기]       │
│  - 의견 카드 공유 버튼 │                         │                  │
│                      │                         │ 모드별 표시:       │
│ [개인 모드]           │                         │ - 팀: 6개 모두     │
│  - 나 ↔ AI 1:1       │                         │ - 개인: 정리·근거  │
│  - "팀에 공유" 버튼   │                         │   확인·질문 만들기 │
├──────────────────────┴─────────────────────────┴──────────────────┤
│ 하단: 모둠 토의 결과                                                 │
│ 대표 의견 | 선택 이유 | 보완할 점 | 실행 계획 | 결과 저장             │
└────────────────────────────────────────────────────────────────────┘
```

**영역별 상세 설명**

- **상단 헤더**: 토의 메타정보 상시 표시. `n/N 접속`은 가변 인원(2-5) 표시.
- **좌측 — 채팅 영역 (탭)**:
  - **팀 탭**: 모둠 전체와 AI 진행자의 공동 토의방. AI는 턴 진행, 근거 요구, 합의 질문을 수행.
  - **개인 탭**: 학생 본인과 AI만 보이는 1:1 사고 확장 공간. 다른 학생은 열람 불가(교사는 사후 조회 가능).
  - 의견 카드는 채팅 영역에서 작성 후 공동 보드로 전송하는 흐름.
- **중앙 — 공동 토의 보드**: 토의의 핵심 작업 공간. 의견 비교표, 선택 기준, 쟁점·공통점·차이점, 모둠 대표 의견 후보, 선택 이유, 실행 방안 입력.
- **우측 — AI 보조 패널**: 학생이 명시적으로 호출하는 6개 버튼 기반.
  - **팀 모드**에서는 6개 모두 활성.
  - **개인 모드**에서는 자기 사고 확장에 해당하는 3개(정리·근거 확인·질문 만들기)만 활성.
- **하단 — 모둠 토의 결과**: 최종 산출물 영역. 대표 의견, 선택 이유, 보완할 점, 실행 계획, 결과 저장 버튼.

**레이아웃 설계 원칙**
1. 좌측 채팅 영역에서 **팀/개인 탭 전환**이 가능하도록 하여, 학생이 흐름을 끊지 않고 모드 간 이동.
2. 개인 모드는 시각적으로 명확히 구분(다른 배경색, "나만 보이는 공간" 표시).
3. AI 패널은 **버튼 기반**으로 구성하여 학생이 능동적으로 도움을 요청하도록 함.
4. 하단 결과 영역은 토의가 진행되며 점진적으로 채워지는 구조 → 진척도 자체가 시각화.

### 4.3 교사 대시보드

| 영역 | 기능 |
|------|------|
| 모둠 진행 현황 | 단계별 진척도, 팀/개인 채팅 활성도, 합의 도달 여부 |
| 개입 알림 | 침묵 모둠, 갈등 모둠, 근거 부족 모둠 표시 |
| 학생별 참여 지표 | 팀 채팅 발언 빈도, 개인 채팅 사용량, 의견 다양성, 동조/반박 패턴 |
| 개인 채팅 사후 조회 | 학생 동의 하에 학생-AI 1:1 대화 열람 (실시간 비노출, 사후만) |
| **팀별 최종 토의 결과** | 모둠별 대표 의견·선택 이유·보완할 점·실행 계획 집계 |

```
│ │모둠    │대표 의견    │선택 이유 │보완할 점 │실행 계획 │제출시각│ │
│ ├────────┼────────────┼──────────┼──────────┼──────────┼────────┤ │

│ │...                                                              │ │
│ └────────┴────────────┴──────────┴──────────┴──────────┴────────┘ │
│ [전체 공유 모드 시작] [CSV 내보내기] [학급 종합 리포트]              │
└────────────────────────────────────────────────────────────────────┘
```

**팀별 최종 토의 결과 — 동작 규칙**:
1. 학생이 `[결과 저장]`을 누르면 `consensus_results`에 기록되고, 교사 대시보드 하단 보드에 **실시간으로 한 행이 추가**된다.
2. 보드는 모둠 제출 순서로 정렬, 미제출 모둠은 회색 행으로 표시.
3. 교사는 행 클릭으로 해당 모둠의 토의 전 과정(팀 채팅 로그, 의견 카드 변화)을 펼쳐볼 수 있다.
4. `[전체 공유 모드 시작]` 클릭 시 학생 화면 하단에 다른 모둠의 대표 의견이 노출되어 전체 토의로 전환된다.

### 4.4 AI 에이전트 기능 — 최종 범위 정의

각 기능에 대해 **허용 범위**와 **제한 범위**를 명확히 분리하여 Agency 보존 원칙을 구조적으로 강제한다.

| AI 기능 | 설명 | 허용 범위 | 제한 범위 |
|---------|------|-----------|-----------|
| **토의 진행** | 시작 안내·턴 관리·근거 요구 | 주제·절차 안내, "○○님 차례입니다", "왜 그렇게 생각하나요?" | 의견에 대한 평가·선택·정답 제시 금지 |
| **개인 코칭** | 개인 채팅에서 사고 확장 | 명료화·반론 자극·근거 보완 질문 | 학생 대신 의견·답안 작성 금지, 다른 학생 의견 누설 금지 |
| **요약** | 나온 의견 정리 | 학생 발언 기반 요약 | 없는 의견 추가 금지 |
| **비교** | 의견 간 공통점·차이점 정리 | 기준 제안 | 대표 의견 선택 금지 |
| **근거 점검** | 의견과 근거 연결 확인 | 보완 질문 제시 | 근거 대신 작성 금지 |
| **질문 생성** | 친구에게 물어볼 질문 제안 | 토의 촉진 질문 | 공격적 반박 질문 금지 |
| **태도 점검** | 존중 표현 안내 | 표현 수정 제안 | 학생 비난 금지 |
| **합의 보조** | 선택 기준과 조정 질문 제시 | 합의 과정 지원 | 최종 결정 금지 |

### 4.5 개입 시점 및 호출 방식

| 기능 | 호출 방식 | 개입 시점 | 채팅 모드 |
|------|-----------|----------|----------|
| 토의 진행 | **AI 자동** | 입장 직후 시작 안내, 매 턴 전환 | 팀 |
| 개인 코칭 | 학생이 개인 탭 진입 | 학생 판단 시 | 개인 |
| 요약 | 학생 버튼 호출 (`[정리하기]`) | 학생 판단 시 | 팀 / 개인 |
| 비교 | 학생 버튼 호출 (`[의견 비교]`) | 의견 카드 2개 이상 등록 후 | 팀 |
| 근거 점검 | 학생 버튼 호출 (`[근거 확인]`) + 의견 등록 직후 자동 점검 | 의견 등록 시 | 팀 / 개인 |
| 질문 생성 | 학생 버튼 호출 (`[질문 만들기]`) + 정체 감지 시 권유 | 토의 정체, 단일 관점 | 팀 / 개인 |
| 태도 점검 | 학생 버튼 호출 (`[태도 점검]`) + 비방 표현 실시간 감지 | 실시간 | 팀 |
| 합의 보조 | 학생 버튼 호출 (`[합의 돕기]`) | 합의 단계 진입 시 | 팀 |

### 4.6 제한 범위(금지 행동) 시스템 프롬프트 가드

각 기능별로 LLM 시스템 프롬프트에 명시적 금지 규칙을 삽입한다:

- **토의 진행 가드**: "의견의 옳고 그름을 평가하지 마라. 다음 차례 학생을 안내하고 근거를 질문 형태로만 요청하라."
- **개인 코칭 가드**: "학생을 대신해 의견이나 답안을 작성하지 마라. 다른 학생의 의견을 누설하지 마라. 명료화·반론·근거 보완 질문만 제공하라."
- **요약 가드**: "학생이 실제로 발화하지 않은 내용을 절대 추가하지 마라. 의역보다 원문 보존을 우선하라."
- **비교 가드**: "어느 의견이 더 낫다고 평가하지 마라. 대표 의견을 추천하거나 선택하지 마라."
- **근거 점검 가드**: "근거를 대신 만들어주지 마라. 학생이 보완할 수 있도록 질문 형태로만 환기하라."
- **질문 생성 가드**: "특정 학생을 공격하거나 비난하는 질문을 만들지 마라. 의견을 향한 호기심 질문만 생성하라."
- **태도 점검 가드**: "학생의 인격을 평가하지 마라. 표현 자체에 대해서만 대안을 제시하라."
- **합의 보조 가드**: "합의문을 직접 작성하지 마라. 합의에 필요한 질문과 정리만 제공하라."

## 5. 설계 원칙 (Design Principles)

### 5.1 Agency 보존 원칙
1. **AI는 의견을 생성하지 않는다.** → 학생 입력을 가공·재구성만.
2. **AI는 답을 정하지 않는다.** → 옵션 제시 또는 질문으로 환원.
3. **AI는 평가하지 않는다.** → "맞다/틀리다" 대신 "이 근거는 어디서 왔나요?" 식 환기.
4. **합의 결정은 학생이 한다.** → AI는 합의안 초안을 제안하지 않고, 학생 발화에서 추출.

### 5.2 상호작용 촉진 원칙
1. **소수 의견 보호**: 다수 의견에 묻힌 발언을 가시화.
2. **차례 보장**: 침묵 학생에게 부드러운 발화 유도.
3. **재진술 권장**: 상대 의견을 자기 말로 다시 말하기 유도.
4. **이유 묻기**: "왜 그렇게 생각해?"를 AI가 대신 묻는 비계.

### 5.3 심리적 안전감 원칙
1. AI 피드백은 **개인이 아닌 모둠 전체** 또는 **익명성** 보장.
2. 부정적 피드백은 **사적 채팅(개인 알림)** 으로 전달.
3. 칭찬/긍정 피드백은 **공개적**으로 표시.
4. "틀린 의견"이라는 표현 금지 → "다른 관점", "검토가 필요한 부분".

### 5.4 교사 권한 우선 원칙
1. 교사는 언제든 AI 개입을 **중지/재개**할 수 있다.
2. 교사는 AI 피드백을 **사전 검토**하고 수정/거부할 수 있다.
3. 모든 자동 메시지는 **로그**로 남아 사후 검토 가능.

## 6. 기능 상세 요구사항 (초안)

### 6.1 요약 기능
- 입력: 최근 N개 발화 + 의견 카드 전체
- 출력: 3-5개 핵심 주장 bullet
- 제약: 학생의 표현을 최대한 보존, 의역 최소화

### 6.2 비교 기능
- 입력: 의견 카드 전체
- 출력: (a) 공통 전제, (b) 분기 지점, (c) 양립 가능 여부
- 시각화: 보드 상에서 의견 카드를 클러스터링

### 6.3 근거 점검 기능
- 체크리스트:
  - 근거가 제시되었는가?
  - 근거가 주장을 실제로 뒷받침하는가?
  - 출처가 확인 가능한가?
  - 사실 vs. 의견을 구분했는가?
- 출력 형식: 학생에게 직접 "보완 질문" 제시

### 6.4 질문 생성 기능
- 유형:
  - **명료화 질문**: "어떤 의미로 사용했어요?"
  - **확장 질문**: "다른 상황에서도 그럴까요?"
  - **반론 자극 질문**: "반대 입장에서는 어떻게 볼까요?"
  - **근거 요구 질문**: "그렇게 생각하는 이유는?"
- 시점: 토의 정체 감지(2분 이상 새 발화 없음), 단일 관점 편향 감지

### 6.5 태도 피드백 기능
- 감지 대상: 비방, 인신공격, 무시 표현, 비속어, 인격 비하
- 비감지 대상: **의견에 대한 정당한 반박** (이는 보호)
- 대응: 1차 비공개 알림 → 2차 모둠 전체 환기 → 3차 교사 알림

### 6.6 합의 보조 기능
- 단계:
  1. 모둠 의견 중 **공통 동의 요소** 추출
  2. **남은 쟁점** 정리
  3. 학생들이 직접 합의안 작성
  4. AI는 합의안의 **완결성 점검**만 수행 (대표성, 근거 포함 여부)
- 금지: AI가 합의 문장 자체를 작성하는 것

## 7. 평가 지표 (Evaluation Metrics)

### 7.1 학습 효과 지표
- 의견 다양성(Diversity Index)
- 근거 포함률(% of claims with evidence)
- 의견 변화 횟수(생각 수정 빈도)
- 합의 도달률 및 도달 시간
- 합의안의 질적 수준 (교사 루브릭 평가)

### 7.2 상호작용 지표
- 학생별 발화 균등도(Gini coefficient of turns)
- 상호 참조율(다른 학생 의견 인용 빈도)
- 비방/부정적 발화 비율
- 침묵 시간

### 7.3 Agency 보존 지표
- AI 발화 대비 학생 발화 비율 (학생 ≫ AI 유지)
- 합의 문장 내 학생 원문 보존 비율
- 학생의 AI 제안 수용/거부 비율 (거부도 정상 신호)
- 학생 설문: "AI가 내 생각을 대신했다고 느꼈는가?"

### 7.4 교사 효용 지표
- 개입 알림 정확도 (false positive 최소화)
- 대시보드 사용 빈도 및 만족도
- 교사 개입 시점과 AI 알림의 일치도

## 8. 윤리·정책 고려사항

### 8.1 데이터·프라이버시
- 학생 발화는 **수업 목적 외 사용 금지**.
- 미성년자 보호자 동의 절차.
- 발화 데이터의 **익명화** 및 **보관 기간** 명시.
- 외부 LLM API 사용 시 데이터 전송 정책 검토.

### 8.2 편향 및 공정성
- AI 요약·비교 시 특정 학생/의견 편향 검증.
- 한국어 청소년 어휘에 대한 모델 성능 확인.
- 방언, 비표준 표현에 대한 비포용성 방지.

### 8.3 교사의 책임 범위
- AI 피드백은 **참고용**임을 학생·교사에게 명시.
- 최종 평가는 **교사**가 수행함을 시스템 곳곳에 명시.
- AI 오류 발생 시 **신고/수정** 채널 제공.

### 8.4 학생 동의 및 투명성
- AI가 어떤 정보를 보는지, 무엇을 하는지 **학생 친화 언어로 공지**.
- AI 발화는 항상 **시각적으로 구분**(아이콘, 색상).

## 9. 기술 아키텍처

### 9.1 추천 기술스택

| 영역 | 추천 기술 |
|------|----------|
| 프론트엔드 | Next.js 15, React, TypeScript |
| UI | Tailwind CSS, shadcn/ui |
| 실시간 동기화 | Supabase Realtime |
| 데이터베이스 | Supabase PostgreSQL |

| 배포 | Vercel |
| 인증 | **Supabase Auth** (회원가입·로그인) + 학생은 방 코드 추가 입장 |
| 권한 | teacher / student role 분리 (Supabase Auth metadata + RLS) |

### 9.2 기술 선택 근거

| 기술 | 선택 이유 |
|------|----------|
| **Next.js 15 + React + TypeScript** | App Router 기반 SSR/CSR 혼합으로 교사 대시보드(SSR)와 토의방(CSR)을 한 코드베이스에서 처리. 타입 안전성으로 의견·근거 데이터 구조 관리. |
| **Tailwind CSS + shadcn/ui** | 빠른 프로토타이핑과 접근성 기본 보장. 학생 대시보드의 3분할 레이아웃을 유틸리티 클래스로 신속 구현. |
| **Supabase Realtime** | 4인 모둠의 의견 카드·공동 보드 실시간 동기화. WebSocket 직접 구현 부담 제거. Postgres Changes 구독으로 의견 추가/수정 즉시 반영. |
| **Supabase PostgreSQL** | 토의 세션, 의견, 근거, 합의안의 관계형 데이터 모델링에 적합. Row Level Security(RLS)로 모둠별 데이터 격리. |
| **OpenAI** | 한국어 청소년 발화 이해도, 응답 latency, 안전 정책을 기준으로 선택. 초기 MVP는 둘 다 추상화 레이어 뒤에 두고 비교 평가 권장. |
| **Vercel** | Next.js와 가장 원활한 배포 통합. Preview Deploy로 교사 파일럿 피드백 빠른 반영. |
| **Supabase Auth** | 교사·학생 모두 회원가입·로그인을 Supabase Auth로 통합. 이메일/비밀번호 + (선택) OAuth. 학생은 로그인 후 방 코드로 모둠 입장. JWT의 `role` 클레임으로 권한 구분. |
| **teacher/student role 분리** | Supabase Auth `user_metadata.role` + RLS 정책으로 백엔드 강제. 교사는 본인 모둠만, 학생은 입장한 모둠만 접근. |

### 9.3 주요 컴포넌트
- **실시간 토의방**: Supabase Realtime 채널 기반 다자 동시 편집 (의견 카드 단위 낙관적 동시성)
- **공동 보드**: 의견 카드 드래그/그룹핑 (DOM 기반, dnd-kit 등)
- **AI 오케스트레이션 레이어**: 발화 누적 → 트리거 판정 → 프롬프트 구성 → LLM 호출 → 후처리 (Next.js API Routes / Edge Functions)
- **모더레이션 레이어**: 비방·부적절 표현 사전 필터 (OpenAI Moderation API 또는 별도 분류기)
- **교사 대시보드**: Supabase Realtime 구독을 통해 모둠 상태 집계 및 알림

### 9.4 데이터 모델 스케치 (Supabase 테이블)

| 테이블 | 주요 컬럼 |
|--------|----------|
| `rooms` | id, room_code, topic, teacher_id, stage, time_limit, min_participants(2), max_participants(5), created_at |
| `participants` | id, room_id, user_id (FK → auth.users), role (teacher/student), nickname, turn_order, joined_at |
| `messages` | id, room_id, channel (`team` / `personal:{participant_id}`), author_id (학생 또는 AI), content, message_type (utterance/ai_facilitation/ai_coaching), created_at |
| `opinions` | id, room_id, author_id, content, evidence, shared_from_personal (bool), created_at, updated_at |
| `board_items` | id, room_id, type (compare/criteria/issue/representative), content, position |
| `ai_messages` | id, room_id, channel (`team`/`personal:{pid}`), feature (facilitation/coaching/summary/compare/check/question/attitude/consensus), content, triggered_by, created_at |
| `consensus_results` | id, room_id, representative_opinion, reason, improvements, action_plan, submitted_at |
| `attitude_flags` | id, room_id, target_participant_id, severity, raw_text, action_taken, created_at |
| `personal_chat_consent` | id, participant_id, room_id, teacher_view_allowed (bool), consented_at |

**채널 분리 핵심 규칙**:
- `messages`와 `ai_messages`의 `channel` 컬럼이 가시성을 결정.
- `team` 채널: 모둠 전원 + 교사 read.
- `personal:{participant_id}` 채널: 본인만 실시간 read, 교사는 `personal_chat_consent.teacher_view_allowed=true`인 경우에 한해 **사후** 조회.

### 9.5 AI 모델 선택 고려사항
- **반응성**: 토의 흐름을 끊지 않을 latency (< 3초 권장)
- **한국어 청소년 발화 이해도**
- **비용**: 다수 모둠 동시 운영 시 token 비용 관리 → 캐싱·요약 압축 전략
- **안전성**: 청소년 대상 콘텐츠 정책
- **추상화**: 모델 변경에 대비해 `aiProvider` 인터페이스로 추상화 (OpenAI/Gemini 교체 가능)

### 9.6 인증·계정 흐름 (Supabase Auth)

**회원가입·로그인은 교사·학생 모두 Supabase Auth 통합**. 학생은 로그인 이후 방 코드로 모둠에 추가 입장한다.

#### 9.6.1 교사 흐름
1. **회원가입**: 이메일 + 비밀번호. 이메일 인증 필수. 학교/소속 입력 (선택).
2. `auth.users.raw_user_meta_data.role = 'teacher'` 설정 (서버 사이드 함수에서 강제).
3. 로그인 후 교사 대시보드로 진입, 방 생성 가능.

#### 9.6.2 학생 흐름
1. **회원가입**: 이메일 + 비밀번호 (보호자 동의 절차 필수, [research.md §8.1](#81-데이터프라이버시)).
2. `auth.users.raw_user_meta_data.role = 'student'`.
3. 로그인 후 **방 코드 입력 화면** → 교사가 배포한 코드 입력 → 닉네임 설정 → 모둠 입장.
4. 입장 시 `participants` 테이블에 `(room_id, user_id, nickname, turn_order)` row 생성.

#### 9.6.3 RLS 정책 연계
- 모든 테이블 RLS는 `auth.uid()`와 `auth.jwt() ->> 'role'`을 사용.
- 교사 권한: `auth.jwt() ->> 'role' = 'teacher'` AND `rooms.teacher_id = auth.uid()`.
- 학생 권한: `participants` 테이블에 `(room_id, user_id = auth.uid())` 존재.

#### 9.6.4 role 부여 강제
- 회원가입 시 클라이언트가 role을 임의로 지정하는 것을 막기 위해, 서버 사이드 Edge Function 또는 Database Trigger로 가입 직후 role 정합성 검증.
- 학생이 임의로 'teacher' role을 갖는 것을 차단.

#### 9.6.5 비밀번호·복구
- Supabase Auth 기본 비밀번호 재설정 이메일 사용.
- 중학생 보호자 연락처 백업 옵션 (선택).
